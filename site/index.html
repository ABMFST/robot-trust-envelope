Robot Trust Envelope · weekend prototype

The cyber control plane for robots wants to look a lot like the one for confidential VMs.

I'm Amir Bredy, a TPM II on Microsoft's Azure Edge Security team. I run Trusted Launch (vTPM + Secure Boot) and own Security Benchmarks for Microsoft's public and regulated cloud environments on Azure Local, across two million-plus devices. This project ports those patterns - hardware-rooted attestation, short-lived signed identity, and CIS-aligned policy gates - onto a simulated robot fleet, then adds two things the cloud doesn't have to think about: STPA-derived runtime safety envelopes and LLM red-team scenarios that try to drive the robot outside those envelopes.

Attestation

An Ed25519 Fleet CA validates a simulated TPM-style quote and issues a short-lived JWT. The robotics analog of Microsoft Azure Attestation + Conditional Access.

2

Safety Envelope

A pure-Python ROS2 node enforces 5 STPA-derived constraints (velocity caps, workspace polygon, obstacle standoff, attestation gate) and overrides /cmd_vel on every violation.

3

LLM Red-Team

An LLM-style operator fires adversarial natural-language goals (speed bonus, leave the cell, ignore the human). The envelope blocks every one.

    Safety method coverage

    The JD lists FMEA, FTA, HAZOP, and STPA. This repo applies STPA to the system (what unsafe actions can the operator or robot take?) and FMEA to the envelope itself (what happens when the safety monitor fails?). Both worksheets are honest learning exercises on a toy system.

    What I'd own in the first 90 days

    What this weekend prototype hints at, scaled to what the role actually owns.

    1. Threat model the production robot fleet end to end. Catalogue assets, trust boundaries, and data flows from device firmware through the cloud control plane. Produce an STPA + STRIDE crossover that engineering, security review, and the safety lead can all sign.
    2. Formalize the safety case. Pick the target standard (ISO 13482 for service robots, ISO 10218 for industrial, IEC 61508 for the underlying SIL claim), staff a formal review with a functional-safety practitioner, and produce a real hazard log and safety requirements traceability matrix.
    3. Stand up a real LLM red-team evaluation harness. Take the toy scenarios in this repo and turn them into a continuous evaluation: prompt corpora, sim-in-the-loop runs, intervention counts as a release gate. Coordinate with Microsoft's Office of Responsible AI on shared methodology.
    4. Define the envelope policy DSL and its control-plane delivery. Modeled on what I already deliver with Azure Guest Configuration and Security Benchmarks: declarative policy, versioned, signed, delivered to every robot, with attested receipt and continuous compliance reporting back. The envelope YAML in this repo is the v0.1 sketch.
    5. Establish the release gate. Bring the same release-gate validation discipline I run today for TVM to the robotics platform: explicit safety + security exit criteria, evidence captured, defect-escape tracked. The repo's CI is the toy version of this.
